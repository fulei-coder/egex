# 台式机1（ROS1 + U-arm leader）→ 台式机2（ROS2 + RealMan 采集/训练）完整改造方案

## 1. 结论

可以这样分机，而且这是更稳的工程方案：

- **台式机1**：只做 **U-arm leader 读取**，运行 ROS1 Noetic 和 `LeRobot-Anything-U-Arm` 的 teleop 输出。
- **台式机2**：继续作为 **唯一的 RealMan 控制机 + 相机采集机 + HDF5 录制机 + 转换/训练机**。

但 **台式机2 的采集程序需要改**。不是改训练主链，而是改 **采集/控制入口**。

核心原则只有一句话：

> **谁控制 RealMan，谁记录 action。**

因此，台式机1 不应该直接控制 RealMan；它只应该把 `leader` 的 7 维动作发送给台式机2。台式机2 收到远端动作后，在本机完成：

1. U-arm → RealMan 映射
2. 限幅/平滑/丢包保护
3. RealMan SDK 下发
4. 本机读取真实 `qpos`
5. 本机记录 `action / qpos / images`

这样 `action` 和 `qpos` 的时间关系最清楚，后续 `convert_to_lerobot.py`、训练和推理链路都能保持不变。

---

## 2. 为什么这样拆是对的

### 2.1 台式机1 的 U-arm 仓库本质是 ROS1 leader-follower 框架

`feat/simulation` 分支的 README 和 `howtoplay.md` 明确写的是：

- 目标环境：**Ubuntu 20.04 + ROS Noetic**
- 原生是 **ROS1**
- leader 输出通过 **`/servo_angles`** 发布
- 默认流程是 `servo_reader.py` → follower 控制脚本 → recorder

所以它天然适合作为 **leader 机**，不适合作为你现在的 **RealMan 采集主进程**。

### 2.2 台式机2 的 RealMan 仓库本质是 SDK 控臂 + 本机采集 + 本机录制

你现在的 `collect_data.py` 已经明确是：

- 顶部相机：本机 `pyrealsense2`
- 腕部相机：本机 ROS2 订阅
- 机械臂：本机 **RealMan Python SDK**
- 录制：同一个进程里写 HDF5

所以台式机2 继续做 **唯一控制机** 是最顺的，不应该为了接 U-arm 把控制权迁到台式机1。

### 2.3 你当前采集脚本还存在一个必须顺手修的点

你贴出来的 `collect_data.py` 当前保存 `action` 的方式不是“真实下发命令”，而是：

```python
qpos = self.data_buffer['qpos']
actions = qpos[1:] + [qpos[-1]]
```

这相当于拿“下一帧状态”近似“当前动作”。

如果改成双机摇操，这个问题会更明显。因为真正的动作来自台式机1，若台式机2 不在本机同步记录“下发给 RealMan 的命令”，后面数据语义会更乱。

所以本次改造中，**必须把 `action` 改成真实 command action**。

## 3. 最终推荐架构

```text
台式机1 (ROS1 / Ubuntu 20.04 / Noetic)
    U-arm leader 串口读取
        └── rosrun uarm servo_reader.py
                └── /servo_angles (Float64MultiArray, 7维)
                        └── remote_uarm_bridge.py
                                └── TCP/UDP/ZeroMQ 发到台式机2

台式机2 (ROS2 / Ubuntu / RTX 3090)
    collect_data_remote_uarm.py
        ├── 接收远端 7维动作
        ├── U-arm -> RealMan 映射
        ├── RealMan SDK 控制
        ├── D435 本机采图
        ├── DS87 ROS2 订阅
        ├── 记录 qpos / action / images / timestamps
        └── 保存 HDF5

后处理
    convert_to_lerobot.py  (保持不变)
    train.sh / configs      (基本不变)
    inference.py            (训练部署时按现有链路使用)
```

我不建议你在两台机之间用 `ros1_bridge` 作为第一版方案。原因不是它不能做，而是：

- 你当前台式机2 主程序并不是 ROS1/ROS2 topic 驱动的机械臂控制框架
- 你真正需要跨机传的只有 **7 维动作量**
- 直接用 socket 协议更短、更好调试、更容易做超时保护

所以第一版建议：

- **台式机1：ROS1 topic 保持原样**
- **台式机1 → 台式机2：直接发 socket**
- **台式机2：继续用 SDK 控臂**

## 5. 台式机2 程序到底要怎么改

下面按“最小侵入、可逐步调试”的方式给你拆。

---

## 5.1 第一步：把 `action` 从伪动作改成真实动作

这是最优先的。

### 当前问题

你现在 recorder 里只存了：

- `qpos`
- `images_top`
- `images_wrist`
- `timestamps`

保存时才临时构造：

```python
actions = qpos[1:] + [qpos[-1]]
```

### 正确改法

增加一个共享命令缓存：

```python
class SharedAction:
    def __init__(self, dim=7):
        self.lock = threading.Lock()
        self.action = np.zeros(dim, dtype=np.float32)

    def set(self, action):
        with self.lock:
            self.action = np.array(action, dtype=np.float32)

    def get(self):
        with self.lock:
            return self.action.copy()
```

然后：

- **控制线程** 每次真正下发 RealMan 命令之前或之后，执行 `shared_action.set(cmd)`
- **录制线程** 每次采样时执行 `cmd = shared_action.get()`，并保存到 `self.data_buffer['action']`

Recorder buffer 改成：

```python
self.data_buffer = {
    'qpos': [],
    'action': [],
    'images_top': [],
    'images_wrist': [],
    'timestamps': []
}
```

保存时直接写：

```python
f.create_dataset('action', data=np.array(self.data_buffer['action'], dtype=np.float32))
```

**这一步做完之后，你的 `convert_to_lerobot.py` 才真正有意义。**

---

## 5.2 第二步：把状态和动作统一成 7 维

`convert_to_lerobot.py` 明确把：

- `observations/qpos` 视为 `(N, 7)`
- `action` 视为 `(N, 7)`
- 第 7 维是 `gripper`

所以台式机2 的采集脚本最好统一成：

- `qpos = [joint1..joint6, gripper]`
- `action = [joint1..joint6, gripper]`

### 需要做什么

#### 真实状态 `qpos`

你现在是：

```python
code, state = self.arm.rm_get_current_arm_state()
joint_angles = state['joint'] if code == 0 else [0] * 6
```

要改成：

```python
joint_q = state['joint'][:6]
gripper_q = cached_gripper_state   # 或从 Modbus 读取后缓存
qpos = np.array(list(joint_q) + [gripper_q], dtype=np.float32)
```

### 重要注意

gripper 不建议每一帧同步阻塞式读 Modbus。

更稳的方案：

- 写夹爪时同步更新 `cached_gripper_state`
- 读状态时优先返回缓存值
- 每隔较低频率再异步刷新一次硬件读数

原因很简单：你自己的 RealMan 文档里已经明确记录了 **Modbus 同步读写会把频率拖到 1–2 Hz**，而异步写 + 缓存读可以恢复到 30 Hz 量级。

## 5.3 第三步：新增远端动作接收器

### 推荐协议

直接用 **TCP**，第一版最稳。

台式机1 每 20 ms 发一包：

```json
{
  "seq": 1024,
  "t_send": 1712345678.123,
  "angles": [a1, a2, a3, a4, a5, a6, g]
}
```

### 为什么选 TCP

- 丢包少
- 实现简单
- 便于日志追踪
- 对 7 维小消息完全够用

### 台式机2 receiver 行为

新增 `RemoteUarmReceiver`：

- 后台线程监听 socket
- 解析 JSON
- 更新 `latest_angles`
- 更新时间戳 `last_recv_time`
- 若超时，例如 `>0.2s`，设置 `is_stale=True`

接口建议：

```python
receiver.get_latest() -> np.ndarray shape (7,)
receiver.is_stale(timeout=0.2) -> bool
```

## 5.4 第四步：新增 U-arm → RealMan 映射器

### 为什么必须单独写 mapper

U-arm leader 输出的是 **leader 相对零位偏移**。

RealMan 侧你真正要发的是：

- 关节绝对目标
- 或围绕某个起始位姿的增量目标

两者之间至少要处理：

1. 关节顺序
2. 正负方向
3. 尺度
4. 零位偏置
5. 夹爪映射
6. 限位保护

### 第一版最稳的策略：增量映射

启动采集时，先读取 RealMan 当前初始关节位：

```python
rm_init_q = robot.get_qpos(include_gripper=True)
```

再把 U-arm 当前输出视为零位，之后远端数据都按偏移映射：

```python
u = receiver.get_latest()          # 7维，单位建议先保持“度”
cmd = rm_init_q.copy()
cmd[:6] = rm_init_q[:6] + sign[:6] * scale[:6] * np.deg2rad(u[:6]) + offset[:6]
cmd[6]  = map_gripper(u[6])
cmd = np.clip(cmd, lower, upper)
```

### gripper 单独处理

不要把第 7 维和前 6 维一起当普通关节矩阵映射。

建议第一版直接做二值夹爪：

```python
cmd[6] = 1.0 if u[6] > threshold else 0.0
```

后面再升级成连续开合。

## 5.5 第五步：控制线程改成“远端 leader 驱动”

你当前 Vive 线程是笛卡尔控制：

- tracker pose
- 算 delta pose
- `rm_movep_canfd(target_6d, ...)`

这套不要复用到 U-arm 上。

### 新控制线程应该这样

新增一个 `RemoteUarmController`：

```python
class RemoteUarmController:
    def __init__(self, arm, arm_lock, receiver, mapper, shared_action):
        ...

    def calibrate(self):
        self.rm_init_q = self.get_robot_qpos()
        self.uarm_zero = self.receiver.get_latest()

    def loop(self):
        while self.running:
            if not self.enabled:
                time.sleep(0.01)
                continue

            if self.receiver.is_stale(0.2):
                continue   # 或进入 hold 模式

            raw = self.receiver.get_latest()
            rel = raw - self.uarm_zero
            cmd = self.mapper.map(rel, self.rm_init_q)
            cmd = self.apply_filter(cmd)

            with self.arm_lock:
                self.send_joint_cmd(cmd[:6])
                self.send_gripper_cmd(cmd[6])

            self.shared_action.set(cmd)
            time.sleep(0.02)
```

### 关键点

- 控制频率可以 30–50 Hz
- 写盘仍然保持 15 Hz
- 控制和录制解耦

## 6.3 新增 bridge 节点

新增 `remote_uarm_bridge.py`，逻辑：

- 订阅 `/servo_angles`
- 连接台式机2: `192.168.x.x:5555`
- 每次收到消息就发 JSON 行文本

伪代码：

```python
#!/usr/bin/env python3
import rospy, socket, json, time
from std_msgs.msg import Float64MultiArray

class Bridge:
    def __init__(self):
        self.host = rospy.get_param('~host', '192.168.1.100')
        self.port = rospy.get_param('~port', 5555)
        self.sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
        self.sock.connect((self.host, self.port))
        self.seq = 0
        rospy.Subscriber('/servo_angles', Float64MultiArray, self.cb)

    def cb(self, msg):
        pkt = {
            'seq': self.seq,
            't_send': time.time(),
            'angles': list(msg.data)
        }
        self.seq += 1
        data = (json.dumps(pkt) + '\n').encode('utf-8')
        self.sock.sendall(data)
```

启动：

```bash
rosrun uarm remote_uarm_bridge.py _host:=192.168.1.88 _port:=5555
```

## 9. 哪些程序不需要改

### 9.1 `convert_to_lerobot.py`

如果你最终保存出的 HDF5 仍然是：

- `observations/qpos` `(N, 7)`

那么这个脚本可以 **不改**。

真正要保证的是：

- 数据质量变干净
- FPS 仍然一致
- task 文本仍然一致

这次双机摇操的改造目标是**采集演示数据**。

`inference.py` 属于训练后的部署链，不需要为了 U-arm 双机采集而改。

4. `rostopic hz /servo_angles` 验证频率

**验收标准**：

- `/servo_angles` 稳定输出 7 维
- 角度变化方向可预期
- 不频繁报回传异常

## 阶段 B：台式机1 → 台式机2 通信打通

1. 台式机2 起 receiver
2. 台式机1 起 bridge
3. 在台式机2 打印收到的 7 维数据

**验收标准**：

- 延迟可接受
- 丢包少
- 断网后超时能检测

## 阶段 C：台式机2 只做远端跟随，不录数据

1. 不启动 recorder
2. 只启动控制线程
3. 做 U-arm → RealMan 关节映射
4. 检查正负方向、比例、限位、夹爪开合

**验收标准**：

- 机械臂跟随方向正确
- 无明显抖动
- 无越界
- 停止摇操时能稳定 hold

## 阶段 D：台式机2 打开录制，但先不转数据

1. 启动相机
2. 开始录一小段 20–30 秒
3. 检查 HDF5 内部：
   - `qpos.shape`
   - `action.shape`
   - 两路图像
   - 时间长度

**验收标准**：

- `qpos` 和 `action` 都是 `(N, 7)`
- `action` 明显是远端下发命令，不再是 `qpos[t+1]`
- 图像和动作长度一致

## 阶段 E：转 LeRobot 数据并抽样检查

```bash
python scripts/convert_to_lerobot.py \
  --input-dir data/raw_hdf5 \
  --output-dir data/pick_cube_15fps \
  --repo-id local/pick_cube_15fps \
  --fps 15 \
  --task "pick up the cube and place it in the basket"
```

检查：

- `observation.state` 是否为 7 维
- `action` 是否为 7 维
- 视频正常

## 阶段 F：再训练

这一阶段才去跑 ACT / Diffusion / Pi0 等训练。

## 11. 关键注意点

## 11.1 只能有一台机器真正控制 RealMan

一定要保证：

- 台式机1 绝不直接控制 RealMan
- 台式机2 是唯一控臂进程

否则会出现抢控制权、动作跳变、记录失真。

## 11.2 控制频率和录制频率不要绑死

建议：

- leader 接收 / SDK 控制：30–50 Hz
- HDF5 录制：15 Hz（与现有流程一致）

## 11.3 采集频率、转换 FPS、推理频率必须一致

你自己的 RealMan 仓库已经把这件事写得非常明确了：

> 采集频率 = 转换 FPS = 推理频率

所以如果你录制仍然是 15 Hz，那么：

- `collect_data_remote_uarm.py` 录制 15
- `convert_to_lerobot.py --fps 15`
- 推理 `--freq 15`

## 11.4 gripper 一定要异步/缓存处理

不要把 gripper 同步阻塞式读写放进主控制循环。

## 11.5 先增量映射，再追求“姿态像不像”

第一版不要追求 U-arm 和 RealMan 完全几何同构。

先追求：

- 可控
- 安全
- 数据稳定
- action 语义正确

后面再优化映射手感。

## 11.6 网络失联保护必须做

建议策略：

- `receiver.last_recv_age() > 0.2s`：停止更新命令
- `> 1.0s`：自动 disable teleop
- 不要把断连理解为发零命令

## 12. 你最终会得到什么

改完以后，你的链路会变成：

- **台式机1**：U-arm leader 专用机
- **台式机2**：RealMan 采集 / 转换 / 训练 / 推理专用机
- 训练链仍然沿用现有 `convert_to_lerobot.py`、训练配置和 `inference.py`
- 真正变化的是：
  - 遥操作输入从 Vive 改成远端 U-arm
  - `action` 从伪造改成真实命令
  - `qpos/action` 统一为 7 维

这是最符合你当前两个仓库实际代码结构的改法。

## 13. 最后给你的实施建议

如果你想把第一次联调成功率拉高，我建议你按下面这个最小可行路径执行：

1. **台式机1 不改 `servo_reader.py` 本体，只新增 bridge 节点**
2. **台式机2 不硬改原 `collect_data.py`，新增 `collect_data_remote_uarm.py`**
3. **先只做远端跟随，不录数据**
4. **确认方向/比例/夹爪都对后，再打开录制**
5. **录制后先人工检查 HDF5 维度和几条轨迹，再转 LeRobot**

这样每一步出问题都能快速定位，不会把 ROS1、ROS2、SDK、相机、录制、训练全搅到一起。
