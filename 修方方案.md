# 用 `LeRobot-Anything-U-Arm` 的 `feat/ros2` 方案摇操 `ros2-lerobot-realman-vla` 的 RealMan RML63：完整改造与部署方案

## 1. 目标与结论

你的目标不是把两个仓库“硬合并”，而是：

- 用 **仓库1** `MINT-SJTU/LeRobot-Anything-U-Arm` 的 `feat/ros2` 分支(/home/tony/lerobot-realman-vla/LeRobot-Anything-U-Arm-feat-ros2)作为 **leader 输入层**；
- 用 **仓库2** `fulei-coder/ros2-lerobot-realman-vla` 继续作为 **RealMan 控制 + 双相机采集 + HDF5 存储 + 转 LeRobot + 训练/推理** 的主干；
- 最终得到一条稳定链路：

```text
U-arm leader  ->  /servo_angles (ROS2)  ->  RealMan RML63 SDK控制  ->  HDF5采集  ->  LeRobot转换  ->  训练  ->  推理
```

**推荐结论：**

1. **直接复用仓库1的 `servo_reader.py` 发布 `/servo_angles`。**
2. **不要复用仓库1的 `servo2xarm.py`、`xarm_pub.py`、`episode_recorder.py`、`cam_pub.py`。**
3. **在仓库2里新增 `U-arm -> RealMan` 的映射控制层。**
4. **机械臂控制继续走 RealMan SDK，不改成纯 ROS2 控臂。**
5. **采集脚本必须改：尤其是 `action` 记录逻辑，不能再用 `qpos[t+1]` 伪造。**
6. **建议把 `qpos` / `action` 统一成 7 维（6关节 + gripper）。**

## 2. 两个仓库里，真正和你这个目标相关的代码是什么

### 2.1 仓库1：`LeRobot-Anything-U-Arm` 的 `feat/ros2`

这条分支从代码上看，确实已经是 **ROS2** 结构，不是仅仅 README 改了字眼。

关键依据：

- `src/uarm/package.xml` 使用 `ament_cmake` / `ament_cmake_python` / `rclpy`；
- `src/uarm/setup.py` 把 `servo_reader`、`episode_recorder`、`xarm_pub`、`servo2xarm` 等注册为 ROS2 console scripts；
- `src/uarm/CMakeLists.txt` 用的是 `find_package(rclpy ...)`、`ament_package()`；
- `src/uarm/scripts/Uarm_teleop/servo_reader.py` 使用 `rclpy.node.Node` 并发布 `/servo_angles`。

### 2.2 仓库1里你真正该用的部分

#### A. `src/uarm/scripts/Uarm_teleop/servo_reader.py`
职责：

- 打开 `/dev/ttyUSB0` 串口；
- 读取 7 路舵机回传；
- 启动时记录 `zero_angles`；
- 做惰性插值平滑；
- 发布 `Float64MultiArray` 到 `/servo_angles`；
- 目标发布速率是 **50 Hz**。

这是你要接入 RealMan 的**最核心可复用输入层**。

#### B. `src/uarm/scripts/Follower_Arm/xarm/servo2xarm.py`
这个文件**不要直接复用**，但非常值得当作参考。

它说明了仓库1作者自己的 follower 设计思路：

- 订阅 `/servo_angles`；
- 把 leader 偏移量映射到 follower 机器人关节；
- 把第 7 维单独映射成 gripper 命令；
- 以 20 Hz 下发 follower 命令；
- 同时把最终执行的 action 发成 `/robot_action`。

你要做 RealMan 版本，本质就是把这份 `servo2xarm.py` 的 follower 思路，改写成 `servo2realman.py` / `UarmLeaderController`。

#### C. `src/uarm/scripts/Follower_Arm/xarm/xarm_pub.py`
这个文件用于把 xArm 当前状态发布为 `/robot_state`。它不适合直接复用，但它告诉你：

- 仓库1把 `state` 和 `action` 区分为两个独立流；
- 采集时它认为 **实际状态** 和 **控制命令** 是两件不同的事。

这个思路对你非常重要。你自己的采集脚本也应该这样做。

#### D. `src/uarm/scripts/episode_recorder.py`
这个 recorder **不要直接用**，因为它：

- 订阅 `/cam_1`、`/cam_2`、`/robot_state`、`/robot_action`；
- 使用 `cv_bridge`；
- 存的是 RLDS / TensorFlow 数据；
- 保存路径和数据格式都不是你当前仓库2的流程。

它更像一个“仓库1自己的采集体系”，不适合拿来替代你当前 HDF5 + `convert_to_lerobot.py` 链路。

### 2.3 仓库2：`ros2-lerobot-realman-vla`

这个仓库的主干现在已经很明确：

- `scripts/collect_data.py`：数据采集主入口；
- `scripts/convert_to_lerobot.py`：HDF5 → LeRobot；
- `scripts/train.sh`：训练入口；
- `scripts/inference.py`：推理执行；
- `hardware/vive_tracker.py`：Vive 输入；
- `docs/troubleshooting.md` / `pipeline_guide.md`：实际部署经验。

### 2.4 仓库2里和这次改造直接相关的事实

#### A. `collect_data.py` 当前是“Vive / teaching”采集脚本
它现在的控制分支只有：

- `ViveController`
- `--teaching`

并且 Vive 控制是 **笛卡尔空间**：

- 读取 Vive pose；
- 计算 `target_pos` / `target_ori`；
- 调 `rm_movep_canfd(target_6d, ...)` 控臂。

也就是说：

- 它不是“关节 leader-follower”结构；
- 它是“末端位姿 teleop”结构；
- 不能把 U-arm 输入直接塞进 `ViveController` 的笛卡尔映射里。

#### B. `collect_data.py` 当前 `action` 是伪造的
`DataRecorder._save()` 里现在是：

```python
qpos = self.data_buffer['qpos']
actions = qpos[1:] + [qpos[-1]]
```

这表示它没有记录**真实控制命令**，而是在保存时用下一帧 `qpos` 近似 `action`。

对于真正的 leader-follower 遥操作采集，这个定义不够严谨，必须改。

#### C. `convert_to_lerobot.py` 假设 `qpos` / `action` 是 7 维
这个脚本的注释和 feature names 都按：

- `joint_1` ... `joint_6`
- `gripper`

来写。

也就是说，你最终最理想的采集格式应该是：

- `observations/qpos`: `(N, 7)`
- `action`: `(N, 7)`

#### D. `inference.py` 里已经有一套很值得复用的 RealMan 控制经验
特别是：

- `RobotController.get_qpos(include_gripper=False/True)`
- `set_qpos()` 里的 EMA 平滑
- 关节死区过滤
- Modbus gripper 异步写入

这份文件说明了一个关键事实：

> **RealMan 夹爪的同步 Modbus 读写很容易把循环拖到 1–2 Hz，所以 gripper 逻辑必须异步化或缓存化。**

这套经验你在 U-arm 遥操作采集里也应该直接沿用。

## 3. 最终建议的总体架构

### 3.1 推荐架构

```text
桌面机2（同一台机器）
│
├─ ROS2节点1：U-arm servo_reader.py
│      └─ 发布 /servo_angles (Float64MultiArray, 7维)
│
├─ ROS2节点2：DS87 相机节点
│      └─ 发布 /Scepter/color/image_raw
│
└─ Python采集主程序（修改后的 collect_data.py）
       ├─ 订阅 /servo_angles
       ├─ 订阅 DS87 图像
       ├─ 本地读取 D435
       ├─ RealMan SDK 控制 RML63
       ├─ 记录真实 qpos/action
       └─ 保存 HDF5
```

### 3.2 为什么不直接拿仓库1整套 recorder/follower 替换仓库2

因为两边主链不兼容：

| 维度 | 仓库1 `feat/ros2` | 仓库2 `ros2-lerobot-realman-vla` |
|---|---|---|
| leader 输入 | `/servo_angles` | Vive / teaching |
| follower 示例 | xArm / Dobot | RealMan SDK |
| 相机 | `/cam_1` `/cam_2` topic | D435 本地 + DS87 ROS2 |
| recorder | RLDS / TF 保存 | HDF5 |
| 训练前格式 | 自己的 recorder 输出 | `convert_to_lerobot.py` 统一转 LeRobot |

如果你强行换成仓库1整套链路，后果是：

- 你得重写相机发布；
- 你得重写 recorder 数据格式；
- 你得改变现有转换、训练与推理流程；
- 你会丢掉仓库2里已经验证过的 RealMan SDK 经验。

所以正确做法不是“替换主链”，而是“只替换 leader 输入方式”。

## 5. 采集程序要怎么改：详细到代码层

## 5.1 保留哪些模块不动

这些现成模块建议保持不变：

- `D435Camera`
- `DS87RosCamera`
- `ensure_ros_node()`
- HDF5 基本结构：
  - `observations/qpos`
  - `observations/images/cam_high`
  - `observations/images/cam_wrist`
  - `action`
  - `timestamps`

也就是说，你的改造中心不是相机，而是：

- teleop 输入
- follower 控制
- action 记录语义

## 5.2 在现有 ROS2 node 上再加一个 `/servo_angles` 订阅

你现在 `collect_data.py` 已经通过 `ensure_ros_node()` 创建了一个全局 `_ros_node`，DS87 的订阅也是挂在这个 node 上。

U-arm 的 `/servo_angles` 订阅也应该复用这个 `_ros_node`，不要再单独 `rclpy.init()` 一次。

建议做法：

```python
class UarmLeaderSubscriber:
    def __init__(self, topic='/servo_angles'):
        ensure_ros_node()
        global _ros_node
        self.lock = threading.Lock()
        self.latest = np.zeros(7, dtype=np.float32)
        self.last_time = 0.0
        self.has_data = False

        self.sub = _ros_node.create_subscription(
            Float64MultiArray,
            topic,
            self._callback,
            10,
        )

    def _callback(self, msg):
        arr = np.asarray(msg.data, dtype=np.float32)
        if arr.shape[0] >= 7:
            with self.lock:
                self.latest[:] = arr[:7]
                self.last_time = time.time()
                self.has_data = True

    def get(self):
        with self.lock:
            return self.latest.copy(), self.last_time, self.has_data
```

## 5.3 新增共享 action 缓存，不再伪造 `action`

这是整次改造中最重要的一步。

你当前 recorder 保存时用：

```python
actions = qpos[1:] + [qpos[-1]]
```

必须改成：

- teleop 控制线程在**每次真正向 RealMan 发指令时**，把命令写进共享缓存；
- recorder 采样时同时读取：
  - 真实 `qpos`
  - 最近一次真实 `action`

建议新增：

```python
class SharedCommandBuffer:
    def __init__(self, dim=7):
        self.lock = threading.Lock()
        self.latest_action = np.zeros(dim, dtype=np.float32)
        self.last_update_time = 0.0

    def set(self, action):
        with self.lock:
            self.latest_action = np.asarray(action, dtype=np.float32).copy()
            self.last_update_time = time.time()

    def get(self):
        with self.lock:
            return self.latest_action.copy(), self.last_update_time
```

然后 `DataRecorder` 改成：

```python
self.data_buffer = {
    'qpos': [],
    'action': [],
    'images_top': [],
    'images_wrist': [],
    'timestamps': [],
}
```

录制时：

```python
actual_qpos = robot_state_reader()
cmd_action, _ = shared_cmd.get()

self.data_buffer['qpos'].append(actual_qpos)
self.data_buffer['action'].append(cmd_action)
```

保存时直接：

```python
f.create_dataset('action', data=np.array(self.data_buffer['action'], dtype=np.float32))
```

不要再用 `qpos[t+1]` 补动作。

## 5.4 `qpos` 要改成 7 维

你现在 `DataRecorder._record_loop()` 里：

```python
code, state = self.arm.rm_get_current_arm_state()
joint_angles = state['joint'] if code == 0 else [0] * 6
```

这只有 6 维。

建议改成：

- 6 维机械臂关节角；
- + 1 维 gripper 状态；
- 合成 7 维 `qpos`。

夹爪状态的读取不要每帧同步阻塞读。建议复用 `inference.py` 的思路：

- 常规情况下读取缓存 `_cached_gripper_pos`；
- 低频刷新 Modbus 读值；
- 或者在采集阶段直接把“最后一次 gripper 命令”作为 qpos 第 7 维近似值。

第一版推荐：

- `action[6]`：使用真实发送的 gripper command；
- `qpos[6]`：优先用缓存 gripper state，实在没有就用最近命令值近似。

## 5.5 新增 `UarmLeaderController`

这就是仓库1里 `servo2xarm.py` 在 RealMan 上的对应物。

建议结构：

```python
class UarmLeaderController:
    def __init__(self, arm, arm_lock, leader_sub, mapper, shared_cmd, robot_iface):
        self.arm = arm
        self.arm_lock = arm_lock
        self.leader_sub = leader_sub
        self.mapper = mapper
        self.shared_cmd = shared_cmd
        self.robot_iface = robot_iface
        self.control_enabled = False
        self.running = True
        self.zero_leader = np.zeros(7, dtype=np.float32)
        self.robot_init = np.zeros(7, dtype=np.float32)
        self.thread = threading.Thread(target=self._loop, daemon=True)
        self.thread.start()

    def calibrate(self):
        leader, _, ok = self.leader_sub.get()
        if not ok:
            print('U-arm leader 当前无数据')
            return False
        self.zero_leader = leader.copy()
        self.robot_init = self.robot_iface.get_qpos(include_gripper=True)
        print('U-arm / RealMan 零位已记录')
        return True

    def enable(self):
        self.control_enabled = True

    def disable(self):
        self.control_enabled = False

    def _loop(self):
        interval = 1.0 / 20.0
        while self.running:
            time.sleep(interval)
            if not self.control_enabled:
                continue

            leader, t, ok = self.leader_sub.get()
            if not ok:
                continue

            action = self.mapper.map(leader, self.zero_leader, self.robot_init)
            self.robot_iface.set_qpos(action)
            self.shared_cmd.set(action)

    def shutdown(self):
        self.running = False
```

### 为什么要这样分层

因为这会让职责非常清晰：

- `leader_sub`：只收 leader 数据；
- `mapper`：只做角度映射；
- `robot_iface`：只和 RealMan SDK 打交道；
- `shared_cmd`：只给 recorder 用；
- `controller`：只负责闭环节奏。

这样后面你要调 joint sign、限幅、gripper 阈值时，不需要碰 recorder。

## 5.6 映射器怎么写

### 关节 1~6

第一版不需要上 IK，直接做“**相对零位偏移 → 关节命令**”。

基本公式：

```python
leader_delta = leader[:6] - zero_leader[:6]
leader_delta = leader_delta[reorder]
leader_delta = leader_delta * sign
leader_delta = leader_delta * scale
cmd_deg = robot_init[:6] + leader_delta + offset_deg
cmd_deg = clip(cmd_deg, lower, upper)
```

### 第 7 维 gripper

仓库1里的 `servo_reader.py` 第 7 维不是普通关节角，不要直接 `np.radians()`。

`servo2xarm.py` 里明确单独做了：

- `self.gripper_range = 0.48`
- `angle_to_gripper()`
- `angle_offset[6] = -angle_offset[6]`

说明这一路是**特殊量纲**，是一个 gripper 开合量，而不是标准角度。

所以你也要单独写：

```python
g = leader[6]
g_norm = (g - leader_open_value) / (leader_close_value - leader_open_value)
g_norm = np.clip(g_norm, 0.0, 1.0)
robot_gripper = robot_open_value + g_norm * (robot_close_value - robot_open_value)
```

第一版可以进一步简化成二值：

```python
robot_gripper = 1.0 if g_norm > 0.5 else 0.0
```

这通常比连续开度更稳。

## 5.7 RealMan 控制接口怎么写

### 关节控制

不要沿用 Vive 的 `rm_movep_canfd()`，因为那是笛卡尔末端控制。

U-arm 方案应该改成 **关节空间控制**。你有两种实现路径：

#### 路径 A：优先推荐
如果你的 RealMan SDK 版本里有“更适合高频关节流式控制”的接口，就用它。

#### 路径 B：当前仓库里已有可靠先例
直接参考 `inference.py`，用：

```python
self.arm.rm_movej(joint_target.tolist(), 5, 0, 0, 0)
```

这条在仓库2现有推理里已经被使用。

第一版建议先用 `rm_movej(...)` 跑通，再考虑换更高频、更平滑的关节流接口。

### gripper 控制

直接复用 `inference.py` 的思路：

- 如果 gripper 指令有变化；
- 单独开线程；
- 用 Modbus 写入；
- 主循环不等它返回。

不要在主控制循环里同步等 gripper 写完。

## 5.8 录制线程建议怎么改

你的录制线程目前只管：

- qpos
- 两路图像
- timestamps

现在应该改成同时存：

- qpos (7)
- action (7)
- cam_high
- cam_wrist
- timestamps

建议在 `_record_loop()` 里：

1. 按固定 `target_fps` 触发一次采样；
2. 读取当前 robot qpos（含 gripper）；
3. 读取当前 shared action；
4. 读取两路图像；
5. 同时 append。

这样 `qpos` 和 `action` 才是同一时刻采下来的。

## 6. 转换、训练、推理这三段还要不要改

## 6.1 `convert_to_lerobot.py`

### 可以不改的条件
只要你采集输出满足：

- `observations/qpos` 是 `(N, 7)`
- `action` 是 `(N, 7)`
- 图像键还是：
  - `observations/images/cam_high`
  - `observations/images/cam_wrist`

那么这个脚本基本可以**不改**。

### 必须注意
这个脚本会把 state/action 的名字固定解释为：

- 6个关节
- 1个 gripper

所以你的 HDF5 必须和这个语义匹配。

## 6.2 训练脚本

`scripts/train.sh` 和各个 `configs/*.yaml` 不需要因为 U-arm 而改。

U-arm 只影响 **采集方式**，不直接影响 LeRobot 训练配置。

## 6.3 `inference.py` 建议改一处

这里有个很重要、很容易被忽略的点：

- `convert_to_lerobot.py` 按 7 维 state/action 设计；
- `inference.py` 当前默认 `robot.get_qpos(include_gripper=False)`，只给模型 6 维 state。

如果你最终训练数据用的是 7 维 `observation.state`，那推理时也应该改成：

```python
qpos = robot.get_qpos(include_gripper=True)
```

否则训练/推理的 state 维度不一致。

### 推荐改法
在 `inference.py` 里增加一个参数：

```bash
--state-with-gripper
```

然后：

```python
qpos = robot.get_qpos(include_gripper=args.state_with_gripper)
```

如果你准备把整个链路统一成 7 维，我建议默认就开启。

## 7.3 `servo_reader.py` 本身建议先补一个小改动

仓库1的 `servo_reader.py` 有两个现实问题：

1. 串口写死为 `/dev/ttyUSB0`；
2. 只把 `baudrate` 做成参数，没有把 `serial_port` 做成参数。

建议你先在仓库1本地改成：

```python
self.declare_parameter("serial_port", "/dev/ttyUSB0")
self.SERIAL_PORT = self.get_parameter("serial_port").get_parameter_value().string_value
```

这样启动时更稳：

```bash

```

如果你不想改源码，也至少要给 U-arm 设备做一个稳定的 udev 规则，避免串口号漂移。

## 8. 联调流程：按什么顺序做，最省时间

## 8.1 阶段 0：只验证 leader 输出

目标：确认 U-arm 串口读数没问题。

检查项：

- `ros2 topic echo /servo_angles --once`
- 转动每个关节，确认对应通道确实变化；
- 第 7 维 gripper 开合时是否变化。

如果这一步都不稳，先不要碰 RealMan。

## 8.2 阶段 1：只做映射，不发机械臂

在 `collect_data_uarm.py` 里先加一个 dry-run 模式：

```bash
--dry-run
```

行为：

- 收 `/servo_angles`
- 做映射
- 打印映射后的 RealMan 目标 `qpos`
- 不发送机械臂命令

检查项：

- 各关节方向是否符合预期；
- gripper 二值阈值是否合理；
- 有没有某一路特别抖。

## 8.3 阶段 2：单关节低速验证

不要一上来 6 关节全开。

做法：

- 在 YAML 里先只开放一个关节 `scale != 0`；
- 其它五个关节 scale 先设 0；
- 缓慢转动 leader 对应关节；
- 观察 RealMan 是否沿正确方向运动。

确认一关后，再依次开第二、第三关。

## 8.4 阶段 3：整机 teleop，不录数据

目标：

- leader-follower 跟随稳定；
- 夹爪可控；
- 主循环频率正常；
- DS87 / D435 图像正常。

检查项：

- 机械臂没有明显跳变；
- 控制线程频率在 15–20 Hz 以上；
- 腕部图像不是黑帧；
- gripper 操作不会把主循环拖到 1–2 Hz。

## 8.5 阶段 4：短时间录制 5–10 秒

录一个最短 episode，然后检查：

```bash
python - <<'PY'
import h5py
f = h5py.File('data/raw_hdf5/pick_cube/task_pick_cube_0.hdf5', 'r')
print('qpos:', f['observations/qpos'].shape)
print('action:', f['action'].shape)
print('cam_high:', f['observations/images/cam_high'].shape)
print('cam_wrist:', f['observations/images/cam_wrist'].shape)
print('timestamps:', f['timestamps'].shape)
PY
```

你要看到：

- `qpos.shape == (N, 7)`
- `action.shape == (N, 7)`

如果不是，就先不要往后走。

## 9. 哪些地方最容易出问题

## 9.1 `servo_reader.py` 第 7 维不是普通关节角

这是最容易踩坑的点之一。

仓库1的 xArm follower 明确把第 7 维当 gripper 开合量处理，不是普通角度。

所以：

- 前 6 维可按关节角思路映射；
- 第 7 维必须单独映射；
- 不要直接对 `leader[6]` 做 `np.radians()` 再加到 RealMan 第 7 维上。

## 9.2 `collect_data.py` 现在强制 15 FPS

原仓库2当前脚本里：

- 传入非 15 的 `--fps` 会被忽略；
- 脚本直接强制 `args.fps = 15`。

如果你想采 30 Hz，需要你自己改。

建议第一版就保持 **15 Hz**，等链路跑稳再考虑提频。

## 9.3 gripper 不能同步阻塞写

仓库2的 `troubleshooting.md` 已经记录过：

- Modbus 同步读写会把循环拖到 1–2 Hz；
- 真正瓶颈通常不在模型，而在硬件通信。

所以在 U-arm 遥操作采集阶段也一样：

- gripper 要异步写；
- 状态最好缓存。

## 9.4 `Float64MultiArray` 没有时间戳

`/servo_angles` 用的是 `Float64MultiArray`，没有 header/timestamp。

同机运行一般不是问题；但如果你后面想跨机，建议换成带时间戳的自定义消息或至少自己封装一层。

## 9.5 训练/推理的 state 维度必须统一

如果训练用的是：

- `observation.state = 7维`

那推理也必须是：

- `robot.get_qpos(include_gripper=True)`

否则模型输入维度和训练统计量对不上。

## 9.6 映射别一开始就追求“完美 1:1”

第一版最稳的目标不是“leader 末端和 follower 末端严格几何对应”，而是：

- 每个关节方向对；
- 动作连续；
- 不抖；
- 不撞限位；
- 能采到干净数据。

也就是说，第一版优先做**关节空间增量映射**，而不是 Cartesian + IK。

## 12. 最终推荐方案（一句话版本）

**用仓库1 `feat/ros2` 的 `servo_reader.py` 发布 `/servo_angles`，在仓库2里新增一个 `U-arm -> RealMan` 的关节空间 follower 控制层；机械臂继续走 RealMan SDK，相机链和 HDF5/LeRobot 训练链保持不变；录制时把 `action` 改成真实发送命令，并把 `qpos/action` 统一成 7 维。**

这条路线侵入最小、复用最高、风险最低，而且最符合你现在桌面机2已经跑通的 RealMan 采集/训练主链。
